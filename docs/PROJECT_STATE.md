## 1. Executive Summary

ArtSoul is an auction-first NFT art protocol on Base. Artwork is published before minting, community discovery and a primary auction establish demand, successful settlement lazily mints the NFT, and indexed on-chain events provide provenance.

The project is in **Phase A: Stabilize Public Testnet**. It is close to a controlled public beta, but Phase A is not complete yet.

Current strengths:

- The production site and public API are online on Vercel.
- The Base Sepolia indexer is online on Hetzner and the public projection was current at audit time.
- The Vite multi-page production build succeeds and all nine built HTML routes pass the build verifier without in-browser Babel.
- The deployed testnet contract covers registration, auctions, deposit-backed bids, settlement, lazy minting, default handling, withdrawals, and resale.
- Focused contract tests pass: 19 of 19.
- Public reads use Supabase projections and caching instead of per-visitor RPC reads.
- The RPC diet from PR #90 is deployed to the Base Sepolia indexer.

Phase A blockers:

- The external-mobile wallet flow is materially improved but still needs the final real-phone acceptance checklist and cleanup of the production debug overlay.
- The general Node regression suite has 20 failing tests out of 132. Several are stale assertions or incomplete browser mocks, but the baseline is not green.
- The public testnet contracts are not mainnet candidates. Resale economics, Genesis behavior, marketplace enforcement, and automatic no-bid finalization do not match the frozen mainnet canon.
- Database migration state, historical secret rotation, and public-repository history cleanup require explicit human verification.
- User-facing complaint submission and a review queue are not implemented. Current moderation is staff hide/unhide only.
- There is no repository CI workflow and no working aggregate `npm test` command.

## 3. Product and Canonical Lifecycle

The canonical user flow is:

1. A creator uploads media and metadata.
2. AI value guidance may be generated by Gemini, but it cannot affect protocol outcomes.
3. The artwork is registered on-chain without minting an NFT.
4. The creator opens a 24, 36, or 48 hour primary auction.
5. Bidders lock the greater of 10% of the bid or 0.01 ETH.
6. Minimum bid increment is the greater of 0.01 ETH or 2.5%.
7. If a winner exists, a 24 hour settlement window opens.
8. Successful settlement mints the NFT to the First Collector and creates the canonical floor.
9. Later resale preserves Creator, First Collector, and current Owner provenance.

Frozen economics:

| Flow | Canon |
| --- | --- |
| Primary sale | 97.5% creator, 2.5% Protocol Treasury |
| Resale | 92.5% seller, 5.5% creator, 1% Protocol Treasury, 1% Ecosystem Pool |
| Defaulted winner deposit | 80% creator, 20% Protocol Treasury |
| Losing bidder deposit | Retains withdrawal rights; no forfeiture |

Trust affects discovery only. It must never affect price, floor, ownership, settlement, royalties, or treasury routing.

## 5. Architecture As Implemented

### 5.1 Frontend and hosting

- Hosting: Vercel
- Build: Vite 8 multi-page application
- UI runtime: React 18 for page entries plus still-global browser modules
- Themes: Classic and Future through `--c-*` variables
- Mobile rule: no idle animation or decorative motion
- Build output: `dist/`
- Serverless API entry: `api/[...route].js`

Built routes:

| Route | Purpose |
| --- | --- |
| `/index.html` | Homepage and Discovery Spotlight |
| `/gallery.html` | Gallery, search, tabs, filters |
| `/artwork.html` | Artwork, auction, discovery, provenance, resale |
| `/profile.html` | Profile and lifecycle tabs |
| `/upload.html` | Publish flow and AI value guidance |
| `/docs-protocol.html` | Protocol documentation |
| `/wallet-test.html` | Isolated wallet diagnostic bench, intentionally not in navigation |
| `/visual-lab.html` | Internal visual test surface |
| `/generate-favicon.html` | Internal favicon utility |

The build migration removed in-browser Babel, but the frontend is still hybrid. Vite bundles the React entries while many classic scripts are copied and loaded as global runtime modules. The successful build emits many warnings that these scripts cannot be bundled without `type="module"`. This is not a deployment failure, but it is performance and dependency-order debt.

### 5.2 API

Deployed Vercel routes include:

- Wallet authentication: nonce, verify, session, logout
- Public projections: artworks, live auction cursor, public config, indexer status
- Discovery writes: like and community signal
- Profile read/update
- OAuth start, callback, and unlink for Discord and X
- Media upload
- Gemini AI analysis
- Artwork and auction compatibility functions
- Staff artwork visibility moderation

Public artwork lists are assembled from Supabase projection tables. A process-local projection snapshot cache defaults to 30 seconds, and list responses are CDN-cacheable. Direct artwork lookups and wallet-specific live auction reads are not shared-cache responses.

### 5.3 Database and storage

- Supabase provides Postgres, Storage, and the public projection backend.
- Protected actions use server-side wallet sessions.
- Media is stored in the public artwork bucket and referenced by projection metadata.
- The repository contains three migration trees: `migrations/`, `sql/migrations/`, and `src/indexer/migrations/`.

There is no single authoritative migration ledger in the repository. Applied migration state must be verified against the linked Supabase database before running any SQL. Migrations must never be replayed blindly.

### 5.4 Indexer

- Production host: Hetzner
- Working directory: `/opt/artsoul`
- Process manager: PM2
- Active process: `artsoul-base-sepolia`
- Health port: 3001
- Stopped legacy process: `artsoul-eth-sepolia`
- Database: Supabase Postgres
- Base public RPC is the primary read endpoint; configured providers are fallbacks

The indexer is incremental. It starts at the last processed block, bounds log ranges, avoids receipt-refetch loops, supports reorg checks, distributed locking, outbox processing, and a disk-backed queue.

Production settings reported after PR #90:

| Setting | Value |
| --- | --- |
| Poll interval | 15,000 ms |
| Maximum block range | 10 |
| Runtime confirmation depth | 3 |
| Reorg check interval | 60,000 ms, which is 60 seconds |
| Reorg sample size | 12 |
| Empty historical block hash backfill | Skipped |

### 5.5 Wallet architecture

Production currently loads:

- `@reown/appkit@1.8.21`
- `@reown/appkit-adapter-wagmi@1.8.21`
- AppKit networks from the same pinned version
- `@walletconnect/ethereum-provider@2.23.10` for the external-mobile core path
- `@walletconnect/modal@2.7.0` for the official mobile WalletConnect modal

Base Sepolia is the only operational write chain. Base mainnet and Ethereum mainnet are present only in WalletConnect negotiation to tolerate wallet namespaces. They are not product networks and writes remain guarded to chain ID 84532. Ethereum Sepolia remains legacy read-only.

External mobile uses a dedicated WalletConnect session path. A persisted core session is the source of truth, modal closure is not destructive, and explicit Disconnect is the only intended teardown action. The isolated `/wallet-test.html` page must remain until production mobile acceptance is complete.

The visual wallet debug overlay is still wired into all six production pages and appears only under its debug flag. Canonical cleanup is pending the final real-phone checklist.

### 5.6 AI value guidance

Gemini 2.5 Flash-Lite is called server-side. The key is not stored in browser local storage. Guidance is optional protocol metadata and cannot alter settlement, floor, royalties, ownership, or mint rights.

### 6.2 Projection status

At audit time the public status endpoint reported:

| Chain | State | Last indexed | Projection lag | Artworks | Auctions | Bids | Settlements | Resale listings |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| Base Sepolia 84532 | running | 44,151,329 | 0 | 24 | 24 | 8 | 5 | 0 |
| Ethereum Sepolia 11155111 | stopped | 10,921,799 | 0 | legacy | legacy | legacy | legacy | legacy |

The public endpoint reported `confirmation_depth=12`, while the founder's direct Hetzner health check after the RPC-diet rollout reported runtime depth 3. This is a real state-reporting defect: `src/indexer/production-runner.js` writes confirmation depth only when inserting a fresh `indexer_state` row and does not update it for an existing row. Runtime polling uses the configured depth, but the persisted status field remains stale. Fix this in a separate code PR and add a regression test.

### 6.3 Hetzner rollout evidence

Founder-run checks after PR #90 showed:

- PM2 process online and listening on port 3001
- Health response `healthy`
- Base Sepolia chain ID 84532
- Five blocks behind during the sampled health call and `isSynced=true`
- 88 indexed events
- Node syntax checks passed
- Focused RPC diet tests passed 3 of 3
- PM2 process list saved

`npm run build` failed on the server because Vite, a development dependency, is not installed there. This does not indicate an indexer runtime failure. Frontend builds belong in CI/Vercel or a complete development checkout, not in the production indexer process.

The optional `failed_events` table never existed: no migration ever created it, its retry marked records resolved before replay, and its replay queue had no consumer, so the gauge reported a constant zero. A-15 (PR #136) retires it instead of creating it. That dead subsystem was masking a real defect: a handler failure rolled back the in-transaction registry claim, the follow-up failed/dead UPDATE matched zero rows, and the cursor advanced anyway, permanently skipping the event while `/health` reported `unresolvedErrors: 0`. `event_processing_registry` is now the single source of truth: failures are UPSERTed outside the rolled-back transaction, a range with any unprocessed event advances no cursor and is retried on the next poll, confirmation is clamped to indexed blocks, and `/health` counts failed/dead rows and reports `degraded`. No new table or migration is required. A one-time advisory-lock release warning was also observed during restart; investigate only if it repeats.
