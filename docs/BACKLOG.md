# ArtSoul Durable Backlog

Updated: 2026-07-16
Canonical phase model: `A-D` from [`docs/canon/ARTSOUL_CANON_BIBLE_FULL.md`](canon/ARTSOUL_CANON_BIBLE_FULL.md) section 17.

This is the repository's durable work register. It consolidates verified completed work, open implementation work, founder-owned inputs, and code-level follow-ups so that ideas do not remain authoritative only in chat sessions or external prompt files.

## Rules

1. The Canon Bible is authoritative for architecture. This backlog schedules and records work; it does not amend canon.
2. Frozen economics may change only through a dated founder amendment. Primary remains `97.5 / 2.5`; resale remains `92.5 / 5.5 / 1 / 1`; all deposit, increment, duration, settlement, and default rules remain those in Bible section 3.
3. Genesis grant cadence, Ecosystem Pool contest triggers, legal entity form and jurisdiction, and launch-readiness windows are tunable or undecided placeholders. A roadmap estimate must never silently become canon, a contract constant, or a treasury rule.
4. Every new product or engineering idea from any chat, report, prompt, review, or partner conversation must first receive a sourced row here before implementation begins.
5. Status is evidence-based: `done`, `in progress`, `planned`, or `blocked-on-founder`. A chat claim alone is not completion evidence.
6. Work follows the canonical phases in order. Phase C contract or premium-status work does not displace unresolved Phase A acceptance blockers.

## Canonical Phase Mapping

| Canon phase | Purpose | Historical planning absorbed |
| --- | --- | --- |
| A - Stabilize Public Testnet | Close production-testnet blockers | Security, wallet, diagnostics, CI, indexer/projection verification, moderation MVP, Base commitments, current UI acceptance |
| B - Public Beta | Validate with real users | Cohort onboarding, incident operations, evidence-backed defect work, durable Snapshot A, build-in-public |
| C - Mainnet Preparation | Build and audit the final system | Contract rework, Genesis, Collections, Ecosystem Pool, admin/eligibility, auras/badges, founder infrastructure, final visual pass |
| D - Staged Mainnet Launch | Migrate and activate safely | Snapshot verification, destructive reset, Base deployment, Core stability window, separate Genesis pilot, controlled growth |

Earlier informal `A-G` plans are retired. Their stabilization/testnet work maps to A, beta validation to B, contract/security/legal/final-product work to C, and migration/deployment/activation to D. Historical token or multichain phases have no mapping and remain out of scope. Source: [`17_ROADMAP_PHASES.md`](canon/17_ROADMAP_PHASES.md) and [`CHANGELOG_2026-07-16_ROADMAP_ALIGNMENT.md`](canon/CHANGELOG_2026-07-16_ROADMAP_ALIGNMENT.md).
